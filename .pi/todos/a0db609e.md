When running `git commit` through bash from the agent, avoid putting backticked command examples or other shell-significant syntax inside double-quoted commit message arguments.

## Why this matters

A few commits succeeded but printed harmless shell warnings because the commit message contained backticks/examples that bash tried to interpret before `git commit` ran.

## Reminder / guideline

- Prefer single-quoted commit message arguments when invoking `git commit` via bash.
- Or avoid backticks/command-like syntax in shell-embedded commit strings.
- Or use a heredoc / commit message file approach for safer multi-paragraph commit messages.

## Goal

Preserve clean commit output and avoid confusing shell warnings during otherwise successful commits.
